# Анализ конвертеров TriG ⇄ Excel (ver2)

Анализ новой в редакции ver2 пары преобразований TriG → Excel и Excel → TriG
(вкладки «TriG → Excel» и «Excel → TriG»). Задействованы модули
[`trig.js`](../js/trig.js) (разбор/сериализация TriG), [`graph.js`](../js/graph.js)
(группировка по графам) и [`trig-workbook.js`](../js/trig-workbook.js) (раскладка по листам).

## Чем TriG отличается от Turtle

TriG добавляет к Turtle **именованные графы**: триплеты группируются в блоки
`graphLabel { ... }`. Триплет в составе именованного графа — это **четвёрка (quad)**
`{ g, s, p, o }`, где `g` — терм-метка графа (или `null` для графа по умолчанию,
то есть триплетов вне фигурных скобок).

Модель TriG в проекте: `{ prefixes, base, quads }`, где `quads` — массив объектов
`{ g, s, p, o }`. Это прямой аналог модели Turtle `{ prefixes, base, triples }`.

## TriG → Excel

1. **Разбор.** `parseTrig(text)` (модуль `trig.js`) переиспользует токенизатор
   `tokenize` из `turtle.js`, добавляя обработку `{`, `}` и необязательного ключевого
   слова `GRAPH`. Метка графа распознаётся по заглядыванию вперёд: терм, за которым
   следует `{`, считается меткой графа.
2. **Группировка по графам.** `buildGraphs(model)` (`graph.js`) собирает квадры в графы
   в порядке первого появления; для каждого графа формирует набор предикатов (столбцов)
   и строки субъектов с ячейками-объектами. Несколько объектов одного предиката в одной
   строке объединяются через запятую.
3. **Раскладка по листам.** `modelToSheets(model)` (`trig-workbook.js`) формирует листы:
   `main`, `Префиксы`, `TriG исходный`, `Квадры простые`, `Графы` и по одному листу на
   каждый граф. На листе `main` в столбце «Предикаты» перечислены предикаты каждого
   листа-графа (требование #3). На листе «Графы» хранится соответствие «лист ⇄ метка графа».
4. **Сборка .xlsx.** `modelToXlsxBytes` собирает книгу и через
   [`xlsx-extras.js`](../js/xlsx-extras.js) добавляет закрепление первой строки и
   автофильтр на каждом листе (требование #1).

## Excel → TriG

1. **Чтение книги.** `workbookToModel` читает листы как AOA.
2. **Источник истины — листы графов.** `sheetsToModel` идёт по строкам листа «Графы»
   и для каждой пары (имя листа, метка графа) читает соответствующий лист-граф:
   из каждой непустой ячейки восстанавливаются квадры. Пустая метка означает граф по
   умолчанию (`g = null`).
3. **Quote/IRI-aware разбиение.** Значения ячейки разбиваются на объекты функцией
   `T.splitObjects`, которая не делит по запятым внутри литералов `"..."` и IRI `<...>`.
4. **Фоллбэк.** Если карта графов отсутствует или пуста, читается лист «Квадры простые».
5. **Сериализация.** `serializeTrig(model)` (`trig.js`) выводит TriG: сначала префиксы,
   затем граф по умолчанию (если есть), затем именованные графы в блоках `g { ... }`.

## Round-trip

Главный инвариант — обратимость: TriG → Excel → TriG не теряет квадров и префиксов.
Проверяется автоматическими тестами в [`tests/test-ver2.js`](../tests/test-ver2.js)
на реальном примере [`examples/trig-vad.trig`](../examples/trig-vad.trig)
(графы `vad:root`, `vad:ptree`, `vad:rtree`, `vad:t_p1`, `vad:t_p1_1`, `vad:t_p2`),
включая проверку через настоящий `.xlsx`.

## Ограничения

- Раскладка по листам не сохраняет порядок квадров внутри графа и форму группировки;
  это семантически эквивалентные сериализации одного RDF Dataset.
- Имена листов Excel ограничены 31 символом и не могут содержать `: \ / ? * [ ]`,
  поэтому метка графа санируется (`sanitizeSheetName`); исходная метка графа при этом
  сохраняется на листе «Графы», что и обеспечивает корректную обратную сборку.
