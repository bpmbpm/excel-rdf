# Альтернативная концепция и сравнение

Основная концепция проекта — «**property table по МегаТипу**»: на каждый МегаТип свой
лист, строки — объекты, столбцы — предикаты. Ниже предложена альтернативная концепция и
дано сравнение плюсов и минусов.

## Основная концепция (МегаТип → лист)

На каждый МегаТип — отдельный лист с широкой таблицей (wide table): строка = субъект,
столбец = предикат, ячейка = объект(ы).

**Плюсы:**
- Наглядно и привычно для пользователя предметной области (похоже на справочник/реестр).
- Удобны фильтрация и сортировка средствами Excel.
- Компактно для однородных данных (много субъектов с одинаковым набором предикатов).

**Минусы:**
- «Разреженность»: если у разных субъектов разный набор предикатов, появляется много
  пустых ячеек.
- Один лист на МегаТип — при множестве типов много листов.
- Многозначные предикаты приходится упаковывать в одну ячейку (через запятую).

## Альтернатива 1. Единая таблица триплетов (long table / EAV)

Одна таблица: `Субъект | Предикат | Объект` (плюс лист префиксов). По сути это лист
«Триплеты простые» как основной и единственный рабочий формат.

**Плюсы:**
- Универсально: вмещает любой RDF-граф без особых случаев (blank nodes, коллекции,
  reification — добавляются столбцами).
- Нет разреженности и проблемы имён листов.
- Простейшее и самое надёжное обратное преобразование.

**Минусы:**
- Менее наглядно для человека: чтобы увидеть «карточку объекта», нужны фильтр/сводная
  таблица.
- Многословно: один объект «размазан» по многим строкам.
- Ручное редактирование «по объекту» менее удобно.

## Альтернатива 2. Свод через сводные таблицы (Pivot)

Хранить данные как long table (Альтернатива 1), а «представления по типу» строить
**сводными таблицами** Excel (PivotTable) поверх неё.

**Плюсы:**
- Один источник истины (long table) + произвольные срезы без дублирования.
- Срезы строятся стандартными средствами Excel, в т.ч. по любому предикату.

**Минусы:**
- Сводные таблицы — только для чтения; правка идёт в long table, что менее наглядно.
- Сложнее объяснить неподготовленному пользователю; зависит от версии Excel.

## Альтернатива 3. JSON/RDF-граф в одной ячейке на объект

На каждого субъекта одна строка, а все его предикаты-значения — в одной ячейке в виде
мини-структуры (JSON/Turtle-фрагмент).

**Плюсы:**
- Одна строка = один объект, компактно, нет разреженности.
- Легко добавлять любые предикаты без новых столбцов.

**Минусы:**
- Теряется главное преимущество таблицы — фильтрация/сортировка по отдельным предикатам.
- Правка структуры в ячейке неудобна и подвержена ошибкам.

## Сводное сравнение

| Критерий | МегаТип → лист (осн.) | Long table (EAV) | Pivot поверх long | JSON в ячейке |
|----------|----------------------|------------------|-------------------|---------------|
| Наглядность «карточки объекта» | Высокая | Низкая | Средняя | Высокая |
| Фильтр/сортировка по предикату | Высокая | Средняя | Высокая | Низкая |
| Универсальность (любой RDF) | Средняя | Высокая | Высокая | Высокая |
| Разреженность (пустые ячейки) | Возможна | Нет | Нет | Нет |
| Простота round-trip | Средняя | Высокая | Высокая | Средняя |
| Удобство ручной правки | Высокая | Средняя | Низкая (правка в источнике) | Низкая |
| Число листов | Растёт с числом типов | 1 | 1 + сводные | 1 |

## Рекомендация

Текущая реализация удачно **сочетает оба подхода**: основной редактируемый формат —
property table по МегаТипу (наглядно и удобно), а лист «Триплеты простые» представляет
тот же граф в виде long table и служит запасным источником для обратного преобразования.
Это даёт наглядность основной концепции и универсальность альтернативы 1 одновременно.

Возможное развитие: добавить генерацию сводных таблиц (Альтернатива 2) поверх листа
«Триплеты простые» для гибких срезов без потери единого источника истины.
