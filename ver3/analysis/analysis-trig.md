# Анализ конвертеров TriG ⇄ Excel (ver3)

Анализ пары преобразований TriG → Excel и Excel → TriG (вкладки «TriG → Excel» и
«Excel → TriG»). Задействованы модули [`trig.js`](../js/trig.js) (разбор/сериализация
TriG, простая и компактная формы), [`trig-megatype.js`](../js/trig-megatype.js)
(раскладка квадров по МегаТипам и сводка по графам) и
[`trig-workbook.js`](../js/trig-workbook.js) (раскладка по листам книги).

## Главное отличие ver3 от ver2

В **ver2** листы анализа TriG строились вокруг **графов**: один лист = один именованный
граф. В **ver3** (см. [issue #5](https://github.com/bpmbpm/excel-rdf/issues/5)) листы
анализа утверждений строятся вокруг **МегаТипов** — так же, как в книге Turtle.

- Для TriG роль предиката «МегаТип» играет `rdf:type` (равнозначно сокращению `a`):
  значение `rdf:type` субъекта задаёт его МегаТип, а значит и лист, на который субъект
  попадает.
- Принадлежность утверждения к именованному графу (TriG) фиксируется **отдельным столбцом
  «TriG»** на листе МегаТипа.
- Строка листа — это **пара (субъект, граф)**: один субъект может встречаться в нескольких
  графах с разными свойствами, и каждая такая комбинация — отдельная строка.

## Чем TriG отличается от Turtle

TriG добавляет к Turtle **именованные графы**: триплеты группируются в блоки
`graphLabel { ... }`. Триплет в составе именованного графа — это **четвёрка (quad)**
`{ g, s, p, o }`, где `g` — терм-метка графа (или `null` для графа по умолчанию,
то есть триплетов вне фигурных скобок).

Модель TriG в проекте: `{ prefixes, base, quads }`, где `quads` — массив объектов
`{ g, s, p, o }`. Это прямой аналог модели Turtle `{ prefixes, base, triples }`.

## TriG → Excel

1. **Разбор.** `parseTrig(text)` (модуль `trig.js`) переиспользует токенизатор
   `tokenize` из `turtle.js`, добавляя обработку `{`, `}` и необязательного ключевого
   слова `GRAPH`. Метка графа распознаётся по заглядыванию вперёд: терм, за которым
   следует `{`, считается меткой графа.
2. **Раскладка по МегаТипам.** `buildTrigMegatypes(model)` (`trig-megatype.js`):
   - для каждого субъекта определяет его МегаТип по первому встреченному `rdf:type`;
   - группирует квадры по МегаТипу субъекта; ключ строки — пара (граф, субъект);
   - формирует упорядоченный список предикатов-столбцов (в том числе сам `rdf:type`).
   Несколько объектов одного предиката в одной строке объединяются через запятую.
   Субъекты без явного `rdf:type` попадают на лист «Прочие».
3. **Сводка по графам.** `buildTrigSummary(model)` (`trig-megatype.js`) для общего листа
   «TriG» собирает по каждому графу его субъекты, предикаты и число квадров.
4. **Раскладка по листам.** `modelToSheets(model)` (`trig-workbook.js`) формирует листы:
   `main`, `Префиксы`, `TriG исходный`, `TriG с Триплеты простые`,
   `TriG с Триплеты компактные`, `Квадры простые`, `TriG` (сводка по графам), `МегаТипы`
   (реестр листов) и по одному листу на каждый МегаТип. Заголовок листа МегаТипа —
   `Субъект | TriG | <предикаты…>`. На листе `main` в столбце «Предикаты» перечислены
   предикаты каждого листа МегаТипа.
5. **Сборка .xlsx.** `modelToXlsxBytes` собирает книгу и через
   [`xlsx-extras.js`](../js/xlsx-extras.js) добавляет закрепление первой строки и
   автофильтр на каждом листе.

## Состав книги TriG (ver3)

1. `main` — перечень листов книги, их назначение и (для листов МегаТипов) их предикаты.
2. `Префиксы` — префиксы исходного TriG.
3. `TriG исходный` — исходный текст TriG.
4. `TriG с Триплеты простые` — TriG, где внутри графов по одному триплету на строку
   (без `;` и `,`).
5. `TriG с Триплеты компактные` — TriG в компактной форме (с `;` и `,`).
6. `Квадры простые` — по одной четвёрке (граф, субъект, предикат, объект) на строку.
7. `TriG` — общий лист: свойства каждого графа TriG (субъекты, предикаты, число квадров).
8. `МегаТипы` — реестр листов МегаТипов (имя листа, терм `rdf:type`, назначение).
9. Листы МегаТипов — на каждый МегаТип свой лист; строка = (субъект, граф).

## Excel → TriG

1. **Чтение книги.** `workbookToModel` читает листы как AOA.
2. **Источник истины — листы МегаТипов.** `sheetsToModel` идёт по реестру «МегаТипы» и
   для каждого перечисленного листа читает строки: субъект из столбца «Субъект», граф из
   столбца «TriG» (пустая ячейка — граф по умолчанию, `g = null`), остальные столбцы —
   предикаты. Поскольку `rdf:type` сохранён как обычный столбец, обратная сборка не теряет
   и не выдумывает квадров `rdf:type`.
3. **Quote/IRI-aware разбиение.** Значения ячейки разбиваются на объекты функцией
   `T.splitObjects`, которая не делит по запятым внутри литералов `"..."` и IRI `<...>`.
4. **Фоллбэки.** Если реестра «МегаТипы» нет — читаются любые листы с сигнатурой заголовка
   `Субъект | TriG`. Если и это не дало квадров — читается лист «Квадры простые».
5. **Сериализация.** `serializeTrig(model)` (`trig.js`) выводит компактный TriG: сначала
   префиксы, затем граф по умолчанию (если есть), затем именованные графы в блоках
   `g { ... }`. `serializeTrigSimple(model)` выводит ту же модель в простой форме
   (по одному триплету на строку, без `;` и `,`).

## Round-trip

Главный инвариант — обратимость: TriG → Excel → TriG не теряет квадров и префиксов.
Проверяется автоматическими тестами в [`tests/test-ver3.js`](../tests/test-ver3.js)
на двух примерах — подробном [`examples/trig-vad.trig`](../examples/trig-vad.trig)
(МегаТипы `vad:ObjectTree`, `vad:TypeProcess`, `vad:TypeExecutor`, `vad:VADProcessDia`,
`vad:ExecutorGroup`; графы `vad:root`, `vad:ptree`, `vad:rtree`, `vad:t_p1`, `vad:t_p1_1`,
`vad:t_p2`) и простом [`examples/trig-simple.trig`](../examples/trig-simple.trig)
(МегаТипы `ex:Product`, `ex:Customer`, `ex:Order`; графы `ex:catalog`, `ex:orders`),
включая проверку через настоящий `.xlsx`.

## Ограничения

- Раскладка по листам не сохраняет порядок квадров внутри графа и форму группировки;
  это семантически эквивалентные сериализации одного RDF Dataset.
- Имена листов Excel ограничены 31 символом и не могут содержать `: \ / ? * [ ]`,
  поэтому имя листа МегаТипа санируется (`sanitizeSheetName`); исходный терм `rdf:type`
  при этом сохраняется на листе «МегаТипы», что и обеспечивает корректную обратную сборку.
- МегаТип субъекта определяется по **первому** встреченному `rdf:type`; если у субъекта
  несколько разных `rdf:type` (в т.ч. в разных графах), он целиком относится к листу
  первого, а все значения `rdf:type` сохраняются в столбце `rdf:type` соответствующих строк.
