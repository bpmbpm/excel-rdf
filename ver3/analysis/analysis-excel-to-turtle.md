# Анализ конвертера «excel to turtle»

Анализ обратного преобразования Excel → Turtle RDF (вкладка «Excel → Turtle»,
функции `workbookToModel` / `sheetsToModel` в [`workbook.js`](../js/workbook.js)
и `serializeTurtle` в [`turtle.js`](../js/turtle.js)).

## Назначение

Собрать корректный Turtle RDF из книги Excel, отредактированной пользователем в удобной
табличной форме (с фильтрацией и сортировкой средствами Excel).

## Конвейер обработки

1. **Чтение книги** (`XLSX.read`) → для каждого листа массив строк (AOA) через
   `sheet_to_json(..., { header: 1 })`.
2. **Сборка триплетов** (`sheetsToModel`):
   - префиксы — с листа «Префиксы»;
   - для каждого листа МегаТипа (первый заголовок = «Субъект»): строится триплет
     `Субъект :МегаТип Тип` и по триплету на каждое непустое значение предиката-столбца;
   - значения с запятыми разбиваются на несколько объектов;
   - лист «Прочие триплеты» добавляется как `Субъект | Предикат | Объект`;
   - если листов МегаТипов нет — используется лист «Триплеты простые».
3. **Сериализация** (`serializeTurtle`) в компактную форму с `;` и `,` и выгрузка `.ttl`.

## Ключевое проектное решение: источник истины

Источник данных для выгрузки — **листы МегаТипов** (и «Прочие триплеты»), а не лист
«Turtle исходный». Это сознательный выбор: именно их редактирует пользователь, и именно
они должны определять результат. Листы «Turtle исходный» / «Триплеты …» носят справочный
характер (хотя «Триплеты простые» используется как запасной источник, если листов
МегаТипов нет).

## Сильные стороны

- **Прямая поддержка пользовательского редактирования** удобной таблицы (а не «сырых»
  триплетов).
- **Корректная обработка списков** объектов в одной ячейке (`:A , :B`).
- **Терпимость к пустым ячейкам** — отсутствующий предикат не порождает «пустых»
  триплетов (соответствует открытому миру RDF).
- **Совместимость по структуре** — читает любую книгу с такой же раскладкой листов, не
  только сгенерированную этим инструментом.

## Ограничения и риски

| Тема | Ограничение | Возможное развитие |
|------|-------------|--------------------|
| Распознавание разделителя объектов | Несколько объектов разделяются запятой; запятая внутри литерала-строки требует, чтобы литерал был в кавычках | Учитывать кавычки при разбиении ячейки |
| Зависимость от заголовков | Лист считается листом МегаТипа по заголовку «Субъект» в первой ячейке | Явный признак листа (служебная пометка/имя диапазона) |
| Терм `:МегаТип` | Предикат принадлежности берётся из заголовка второго столбца | Конфигурируемый предикат МегаТипа |
| Валидация ввода | На стороне JS выполняется минимальная очистка (trim, пропуск пустых) | Перенести сюда часть проверок из VBA для предупреждений в браузере |
| Конфликты правок | Если пользователь правит и таблицы, и лист «Триплеты простые», приоритет у таблиц | Документировать и/или синхронизировать листы средствами VBA |

## Согласованность с проверками VBA

Инварианты, проверяемые VBA в Excel (пустой субъект, дубликат субъекта/предиката,
корректность терма — см. [`doc/vba-checks.md`](../doc/vba-checks.md)), согласованы с
логикой сборки: корректно отредактированная книга без предупреждений даёт корректный
Turtle. JS-сборка дополнительно «прощает» пустые ячейки и лишние пробелы.

## Вывод

Обратный конвертер реализует ключевой сценарий проекта — правка модели в Excel и выгрузка
обратно в Turtle — с сохранением триплетов и префиксов. Риски связаны в основном с
разбором содержимого ячеек (запятые внутри литералов) и распознаванием листов; они
устранимы и не мешают типовому использованию.
